# Reproducibility Map

Every result this codebase can produce is generated by a script, from data
this codebase can load, with no manual spreadsheet editing. This document
maps each kind of result to the exact command that produces it, and what
that command needs (API keys, prior pipeline stages, human annotation).

None of the commands below have been run against the real OpenAI/Anthropic
APIs as part of building this codebase — every module is unit-tested
offline against deterministic fakes (see `tests/unit/`, `tests/fakes.py`),
but producing real numbers requires your own API keys and compute (see
`docs/HARDWARE_SOFTWARE_REQUIREMENTS.md`). Treat every command below as
verified-to-run, not verified-to-produce-a-specific-number.

## 1. Setup

```bash
pip install -r requirements.txt -r requirements-dev.txt
cp .env.example .env   # fill in OPENAI_API_KEY at minimum
pytest tests/unit -q   # confirm the offline suite is green before running anything real
```

## 2. Dataset and claim statistics

| Produces | Command | Needs |
|---|---|---|
| Dataset/claim statistics table (per-dataset Q/A count, generated-answer count, claim count, verdict distribution, manual-label count, path-label count) | `python experiments/build_dataset_statistics.py --datasets hotpotqa,2wikimultihopqa,musique,fever,ragtruth --split validation` | `--predictions_dir`/`--annotations_dir` for non-zero claim/label columns (see below) |

## 3. Main claim-level verification results

| Produces | Command | Needs |
|---|---|---|
| Per-method claim accuracy, per-class F1, macro-F1, path correctness, ECE, with bootstrap CIs (all 11 methods) | `python experiments/run_main_verification_benchmark.py --datasets <list> --split validation --max_samples <N>` | `OPENAI_API_KEY` (or `--llm_backend local`) |
| Per-dataset breakdown + strongest-baseline significance test | `python experiments/run_per_dataset_comparison.py --datasets <list>` | same |

Both scripts write per-method prediction files
(`<output_dir>/<dataset>/<method>_predictions.json`) that every downstream
analysis script below reads instead of re-running verification.

## 4. Retrieval-noise stress test

```bash
python experiments/run_retrieval_noise_stress_test.py \
    --datasets hotpotqa,2wikimultihopqa --methods graphverify_hybrid,llm_text_verifier
```
Needs the same generated-answer/API prerequisites as section 3.

## 5. Component ablation

```bash
python experiments/run_component_ablation.py --dataset hotpotqa --split validation
```
Delegates to `eval/ablation.py`; every variant maps to a real
`GraphVerifyConfig` field (see `eval/ablation.py`'s module docstring) and
`run_variant` raises if a variant ever references a field that doesn't
exist, so this table can never silently regress to a no-op ablation again.

## 6. Contradiction-type breakdown

```bash
python experiments/run_contradiction_taxonomy_breakdown.py \
    --predictions_dir output/results/main_benchmark \
    --datasets hotpotqa,fever --methods graphverify_hybrid,graphverify_score
```
**Needs** section 3's predictions to already exist at `--predictions_dir`.

## 7. Qualitative evidence-path examples

```bash
python experiments/collect_qualitative_examples.py \
    --predictions_dir output/results/main_benchmark \
    --dataset hotpotqa --method graphverify_hybrid
```
**Needs** section 3's predictions.

## 8. Label efficiency, threshold transfer, threshold sensitivity

| Produces | Command |
|---|---|
| Accuracy vs. % of dev labels used for threshold selection | `python experiments/run_label_efficiency_experiment.py --dataset hotpotqa` |
| Thresholds tuned on one dataset, evaluated on others | `python experiments/run_cross_dataset_threshold_transfer.py --tune_on hotpotqa --eval_on 2wikimultihopqa,musique,fever,ragtruth` |
| Full (support_threshold, contradict_threshold) grid | `python experiments/run_threshold_sensitivity_sweep.py --dataset hotpotqa` |

All three run the pipeline once and re-derive verdicts from cached
`support_score`/`contradict_score` for every threshold candidate (see
`graphverify/verdict_assigner.py:VerdictAssigner.verdict_from_scores`), so
they are cheap relative to section 3 despite covering many operating points.

## 9. Oracle pipeline decomposition

```bash
python experiments/run_oracle_pipeline_decomposition.py --dataset hotpotqa --data_dir <dir with gold_claims/gold_graph fields>
```
**Needs** a dataset file where samples carry `gold_claims`/`gold_graph`
(produced by merging real human annotation back into the sample dict — see
section 11). Without these fields, every variant reports the same
full-pipeline number and `n_gold_claims_available`/`n_gold_graph_available`
will be 0 — check those columns before trusting an "oracle" row as
meaningful.

## 10. Runtime and compute

```bash
python experiments/benchmark_runtime_and_compute.py --dataset hotpotqa --max_samples 50
```
No prerequisites beyond API access; produces the numbers
`docs/HARDWARE_SOFTWARE_REQUIREMENTS.md`'s cost table is based on.

## 11. Human annotation agreement

This is the one category of result this codebase cannot produce on its
own: it requires real annotators following `docs/ANNOTATION_GUIDELINES.md`
to produce CSV files in the `dataset/claim_annotation.py` schema.

```bash
python experiments/compute_annotation_agreement.py \
    --annotator_files annotator1.csv,annotator2.csv \
    --adjudicator_file adjudicator_rulings.csv \
    --adjudicated_output output/annotations/adjudicated.csv
```
Outputs Cohen's kappa and Krippendorff's alpha for verdict labels and
path-correctness labels **separately**, plus an adjudicated label file that
`run_oracle_pipeline_decomposition.py` and `build_dataset_statistics.py`
can consume.

## 12. Generator / retriever transfer, long-form evaluation, answer-level reliability

| Produces | Command |
|---|---|
| Metrics across 2+ generator backbones | `python experiments/run_generator_transfer.py --dataset hotpotqa --generators openai:gpt-4o-mini,anthropic:claude-3-5-haiku-20241022` |
| Metrics across 2+ embedding-model retrievers (re-ranking the existing candidate pool — see the script's docstring for the exact scope) | `python experiments/run_retriever_transfer.py --dataset hotpotqa --retrievers BAAI/bge-base-en-v1.5,facebook/contriever` |
| RAGTruth long-form (Summary/Data2txt) hallucination detection | `python experiments/run_longform_citation_evaluation.py --split test` |
| Coverage/abstention (accepted %, EM/F1) across reliability thresholds | `python experiments/run_answer_level_reliability.py --dataset hotpotqa --method graphverify_hybrid` |

## Prompts and configuration

Every prompt any script above sends to an LLM lives as a plain-text file
under `prompts/` (see `graphverify/prompts.py`). Every hyperparameter
(thresholds, weights, matching tiers, evidence composition/source) lives in
`graphverify/config.py:GraphVerifyConfig`, whose class docstring documents
exactly what each field controls. There is no hidden configuration.
